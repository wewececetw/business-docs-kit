# 商業邏輯文件規範

這個目錄存放專案的商業邏輯文件。每份檔案描述一個功能「為什麼存在」「怎麼運作」「為什麼這樣設計」。

## 10 條寫作規則

### R1. 一句話開頭
最上面一定有一句話講完這是什麼。不讀完整份也知道大概。

### R2. Why > What
Code 已經說了 what，文件只寫 why（動機、決策理由、不做什麼的理由）。

### R3. 用人話不用術語
用具體例子取代抽象名詞。
- 好：「爸爸手機看不到媽媽的收藏」
- 壞：「device_id 隔離」

### R4. 固定六節
每份文件固定六個小節（缺可省，不亂加）：
1. 為什麼需要這個
2. 使用者怎麼用
3. 關鍵規則
4. 為什麼這樣設計
5. 會碰到的狀況
6. （選填）不做什麼

### R5. 規則配例子
寫規則時一定附一個具體例子。沒有例子的規則就刪掉。

### R6. 不重複 code
禁止貼 API schema、DB table 結構、code snippet。那些查 code 就好。

### R7. 決策要留 why
關鍵技術選型、架構決策要留「為什麼選這個、為什麼不選那個」。

### R8. 更新日期要同步
功能變更 → 文件更新 → 改「最後更新」日期。

### R9. 每份檔對應一個功能
不跨功能混寫。`songbook.md` 只寫歌本，不寫候播。

### R10. 不做什麼也要寫
「為什麼不做 A」跟「為什麼做 B」同等重要。避免未來反覆討論。

## 如何建立新文件

在 Claude Code 裡執行：
```
/new-doc <功能名>
```
會自動從 `_template.md` 產生骨架。
